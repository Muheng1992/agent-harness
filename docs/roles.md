# Agent Harness — 角色系統詳細說明

> 11 個內建角色的設計理念、適用場景、工具權限與範例 prompt。

---

## 角色系統概述

Agent Harness 的多角色系統讓每個任務都由最適合的「專家」來執行。角色定義檔位於 `roles/` 目錄，採用 Markdown + YAML frontmatter 格式，包含：

- **系統 prompt**：定義角色的身分、行為準則、輸出格式
- **工具權限**：限制角色可使用的 Claude Code 工具，實現最小權限原則
- **模型設定**：可針對不同角色指定模型（目前統一使用 `inherit`）

### 權限分類

角色依工具權限分為三類：

| 類別 | 特徵 | 角色 |
|------|------|------|
| **分析型**（只讀） | 只能讀取、搜尋、執行指令，不能修改檔案 | planner, architect, reviewer, security-auditor |
| **實作型**（讀寫） | 可以建立、修改檔案 | implementer, tester, debugger, documenter, integrator, devops |
| **研究型**（只讀+網路） | 可以上網搜尋，但不修改檔案 | researcher |

## 1. planner — 任務規劃師

### 設計理念

任務規劃師是整個工作流程的起點。在動手寫程式碼之前，先充分理解需求並拆解為可執行的原子任務，定義清楚的依賴關係與完成條件。這避免了「邊做邊想」導致的返工。

### 適用場景

- 收到複雜需求，需要拆解為多個子任務
- 建立新專案的任務 DAG
- 評估任務優先順序與並行可能性
- 在 pipeline 中作為前置分析步驟

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Read | 允許 | 需要閱讀現有程式碼理解架構 |
| Grep | 允許 | 搜尋程式碼以評估影響範圍 |
| Glob | 允許 | 瀏覽專案結構 |
| Bash | 允許 | 執行分析指令（如 `wc -l`、`git log`） |
| Edit | **禁止** | 規劃師不修改程式碼 |
| Write | **禁止** | 規劃師不建立檔案 |

### 範例 prompt

```
為「使用者通知系統」功能進行任務拆解。分析現有程式碼結構，定義需要新增的模組、
介面與測試，並建立任務間的依賴關係。每個任務指定建議的執行角色。
```

## 2. researcher — 線上研究員

### 設計理念

在實作之前先做功課。研究員透過網路搜尋最新的技術方案、最佳實踐與已知陷阱，避免團隊重複踩坑。研究結果以結構化報告呈現，供後續角色參考。

### 適用場景

- 需要評估多個技術方案的優劣
- 查詢特定 API 或 SDK 的用法
- 研究業界最佳實踐
- 在 `full-dev-cycle` pipeline 的第一個階段

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Read | 允許 | 閱讀現有程式碼以理解需求背景 |
| Bash | 允許 | 執行輔助指令 |
| WebSearch | 允許 | 上網搜尋技術資訊 |
| WebFetch | 允許 | 讀取網頁內容（文件、教學） |
| Grep | 允許 | 搜尋程式碼 |
| Glob | 允許 | 瀏覽檔案結構 |
| Edit | **禁止** | 研究員不修改程式碼 |
| Write | **禁止** | 研究員不建立檔案 |

### 範例 prompt

```
上網搜尋 OAuth 2.0 + PKCE 在 iOS App 中的最佳實踐、常見方案、已知陷阱。
比較 ASWebAuthenticationSession 與自建 WebView 的優缺點。輸出結構化研究報告。
```

## 3. architect — 系統架構師

### 設計理念

架構師負責「畫藍圖」，定義系統的高階結構、模組邊界與介面契約。只產出設計文件，不直接寫程式碼——這確保設計決策經過充分思考而非邊寫邊改。

### 適用場景

- 設計新功能的系統架構
- 定義模組間的 API 介面與資料模型
- 評估不同技術方案的架構影響
- 在 `full-dev-cycle` pipeline 的設計階段

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Read | 允許 | 需要深入閱讀現有架構 |
| Grep | 允許 | 搜尋現有 Protocol/Interface 定義 |
| Glob | 允許 | 理解專案目錄結構 |
| Bash | 允許 | 執行分析指令（如查看依賴圖） |
| Edit | **禁止** | 架構師不直接修改程式碼 |
| Write | **禁止** | 架構師不建立程式碼檔案 |

### 範例 prompt

```
為「購物車」功能設計系統架構。考慮現有 Clean Architecture 分層，定義 Domain 層的
Entity 與 Use Case Protocol、Data 層的 Repository 實作、Presentation 層的 ViewModel。
輸出元件圖、介面定義與資料流。
```

## 4. implementer — 程式碼實作專家

### 設計理念

實作專家是團隊的「雙手」，將設計轉化為可運作的程式碼。核心原則是「先讀後寫」——理解現有風格後再動手，確保新程式碼與專案風格一致。嚴格遵循最小變更原則，不做額外重構。

### 適用場景

- 實作新功能
- 根據架構設計撰寫程式碼
- 套用修復方案（當 debugger 角色不可用時）
- 在多數 pipeline 中擔任實作階段

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Edit | 允許 | 修改現有檔案 |
| Write | 允許 | 建立新檔案 |
| Bash | 允許 | 執行建置、測試指令 |
| Read | 允許 | 閱讀設計文件與現有程式碼 |
| Grep | 允許 | 搜尋相關程式碼 |
| Glob | 允許 | 瀏覽檔案結構 |

### 範例 prompt

```
根據架構設計實作使用者登入功能。遵循專案既有的 Clean Architecture 風格。
在 Domain 層定義 LoginUseCase，Data 層實作 AuthRepository，
Presentation 層建立 LoginViewModel 與 LoginView。
```

## 5. tester — 測試工程師

### 設計理念

測試工程師確保程式碼的正確性與可靠性。嚴格遵循 AAA 模式（Arrange → Act → Assert），每個測試只驗證一個具體行為。除了 happy path，特別關注邊界條件與異常輸入。

### 適用場景

- 為新功能撰寫單元測試與整合測試
- TDD 流程中先寫測試（Red 階段）
- Bug 修復後撰寫回歸測試
- 在 pipeline 中負責測試階段

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Edit | 允許 | 修改測試檔案 |
| Write | 允許 | 建立新測試檔案 |
| Bash | 允許 | 執行測試指令 |
| Read | 允許 | 閱讀待測程式碼 |
| Grep | 允許 | 搜尋測試相關程式碼 |
| Glob | 允許 | 瀏覽測試目錄結構 |

### 範例 prompt

```
為購物車計價邏輯撰寫完整測試。涵蓋：正常計價、空購物車、折扣碼套用、
超過庫存數量、負數數量處理。使用專案的 Swift Testing 框架（@Test, #expect）。
```

## 6. reviewer — 程式碼審查員

### 設計理念

審查員以建設性、具體、可操作的方式提供審查意見。每個問題都附上具體的檔案位置與修改建議，並按嚴重度分級（CRITICAL / HIGH / MEDIUM / LOW）。同時也會肯定好的做法。

### 適用場景

- 審查新功能的程式碼品質
- 在 pipeline 的審查階段
- 檢查程式碼是否符合專案架構慣例
- 識別潛在的效能問題或 bug

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Read | 允許 | 需要仔細閱讀待審查的程式碼 |
| Grep | 允許 | 搜尋相關程式碼模式 |
| Glob | 允許 | 確認檔案結構合規 |
| Bash | 允許 | 執行靜態分析、lint 指令 |
| Edit | **禁止** | 審查員不直接修改程式碼 |
| Write | **禁止** | 審查員不建立檔案 |

### 範例 prompt

```
審查使用者登入功能的程式碼變更。檢查：命名一致性、架構分層合規性、
錯誤處理完整性、效能考量、安全性（密碼處理、token 儲存）。
輸出結構化審查報告。
```

## 7. debugger — 除錯專家

### 設計理念

除錯專家使用 5 Whys 分析法，從錯誤訊息出發層層追蹤至根因。核心原則是「先診斷再修復」——不看到錯誤就直覺修改，而是完成根因分析後再套用最小修復。這是 Pipeline Fix Loop 中的關鍵角色。

### 適用場景

- Pipeline 的 fix loop 中（測試失敗、審查失敗）
- `bug-fix` pipeline 的除錯階段
- 任何需要根因分析的場景
- 編譯錯誤與執行時期錯誤的排查

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Edit | 允許 | 修復 bug 需要修改程式碼 |
| Write | 允許 | 可能需要建立修復檔案 |
| Bash | 允許 | 執行測試、查看 log |
| Read | 允許 | 閱讀錯誤相關程式碼 |
| Grep | 允許 | 搜尋錯誤模式 |
| Glob | 允許 | 定位問題檔案 |

### 範例 prompt

```
回歸測試失敗，錯誤訊息如下：
"Expected status 200 but got 401 for POST /api/cart/checkout"
使用 5 Whys 分析法追蹤根因並修復。確保修復範圍最小化。
```

## 8. security-auditor — 安全稽核員

### 設計理念

安全稽核員按照 OWASP Top 10 的順序進行系統性掃描，確保不遺漏常見安全漏洞。以零誤報為目標，只報告有足夠信心的問題，不確定的標註為「疑似」。每個發現都附上可重現的攻擊路徑與修復建議。

### 適用場景

- `security-review` pipeline 的掃描階段
- 上線前的安全審查
- 新增認證/授權功能後的安全驗證
- 處理敏感資料（支付、個資）的模組審查

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Read | 允許 | 閱讀原始碼進行安全分析 |
| Grep | 允許 | 搜尋潛在的安全模式（hardcoded secrets、eval 等） |
| Glob | 允許 | 掃描檔案結構（.env、config 等） |
| Bash | 允許 | 執行安全掃描工具 |
| Edit | **禁止** | 稽核員不修改程式碼（修復由 implementer 執行） |
| Write | **禁止** | 稽核員不建立檔案 |

### 範例 prompt

```
對認證模組進行全面安全掃描。檢查 OWASP Top 10、硬編碼密鑰、注入漏洞、
認證缺陷、權限問題。針對 iOS 本地 App 調整重點至 Keychain 使用、
Certificate Pinning、本地資料加密。輸出安全審查報告。
```

## 9. documenter — 文件撰寫員

### 設計理念

好的文件解釋的是 WHY（為什麼這樣做）而非 WHAT（做了什麼——程式碼本身就說明了）。文件撰寫員站在「第一次接觸專案的工程師」的角度撰寫，確保文件實用且可操作。

### 適用場景

- `full-dev-cycle` pipeline 的文件階段
- 新功能上線後更新 README 和 API 文件
- 撰寫架構設計文件（ADR）
- 為複雜邏輯加上 inline 註解

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Write | 允許 | 建立新文件檔案 |
| Edit | 允許 | 修改現有文件 |
| Read | 允許 | 閱讀程式碼以確保文件準確 |
| Grep | 允許 | 搜尋相關文件與程式碼 |
| Glob | 允許 | 瀏覽文件目錄結構 |
| Bash | 允許 | 驗證文件中的指令範例可執行 |

### 範例 prompt

```
撰寫使用者通知系統的使用文件和 API 文件。包含：功能概述、快速開始指南、
API 介面說明（含參數和回傳值）、使用範例（可直接執行）、常見問題。
```

## 10. integrator — 整合專家

### 設計理念

當多個 agent 在不同 worktree 中並行工作時，需要有人負責將成果合併為一致的整體。整合專家理解每個 agent 的意圖，在語義層面解決衝突，而非僅做文字層面的合併。

### 適用場景

- 並行開發後合併多個 worktree 的成果
- 解決 git merge conflict
- 跨模組的介面相容性驗證
- 端對端整合測試

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Edit | 允許 | 解決合併衝突需修改檔案 |
| Write | 允許 | 可能需建立整合配置 |
| Bash | 允許 | 執行 git merge、建置、測試 |
| Read | 允許 | 閱讀各分支的變更內容 |
| Grep | 允許 | 搜尋跨模組的介面使用 |
| Glob | 允許 | 瀏覽衝突檔案 |

### 範例 prompt

```
合併 Worker 1（auth 模組）和 Worker 2（cart 模組）的工作成果。
兩者都修改了 AppRouter.swift 和 DIContainer.swift，需要語義合併。
確保合併後所有測試通過。
```

## 11. devops — DevOps 工程師

### 設計理念

DevOps 工程師讓開發流程自動化、可重現、可靠。所有配置必須版本化，secrets 絕不寫死在設定檔中。CI Pipeline 設計以快速回饋為目標，將慢速測試放在後段。

### 適用場景

- 設計與實作 CI/CD Pipeline
- 撰寫 Dockerfile 與容器化配置
- 建置自動化腳本
- 環境配置管理

### 工具權限

| 工具 | 權限 | 理由 |
|------|------|------|
| Edit | 允許 | 修改 CI 配置、Dockerfile |
| Write | 允許 | 建立部署腳本、配置檔 |
| Bash | 允許 | 測試 Pipeline、建置容器 |
| Read | 允許 | 閱讀現有配置 |
| Grep | 允許 | 搜尋配置模式 |
| Glob | 允許 | 瀏覽配置檔結構 |

### 範例 prompt

```
為專案設計 GitHub Actions CI/CD Pipeline。包含：lint → build → test → deploy-staging。
使用多階段 Docker 建置減小映像大小。所有 secrets 使用 GitHub Secrets 管理。
```

## 如何選擇角色

### 依任務類型選擇

| 任務類型 | 建議角色 | 建議 Pipeline |
|----------|---------|---------------|
| 需求分析與拆解 | `planner` | — |
| 技術調研 | `researcher` | — |
| 系統設計 | `architect` | — |
| 功能實作 | `implementer` | `quick-implement` |
| 撰寫測試 | `tester` | `tdd-cycle` |
| 程式碼審查 | `reviewer` | — |
| Bug 修復 | `debugger` | `bug-fix` |
| 安全檢查 | `security-auditor` | `security-review` |
| 撰寫文件 | `documenter` | — |
| 合併成果 | `integrator` | — |
| CI/CD 配置 | `devops` | — |
| 完整功能開發 | — | `full-dev-cycle` |

### 單任務 vs Pipeline

- **單任務**：簡單、獨立的工作，直接在 task JSON 中指定 `role`
- **Pipeline**：多步驟的工作流程，使用 `pipeline.py create` 自動產生任務鏈
